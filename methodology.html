Methodology

Overview

4 AI models were given the exact same prompt for each of 9 coding tasks. Each model produced a single self-contained HTML file. No edits, no fixes, no follow-up prompts. The raw output is what you see on this site.

No Claude Code was used to generate any results. Claude Code is Anthropic's agentic coding CLI -- it can read files, run commands, and iterate. That would be an unfair advantage. Instead, each model was called via a single-shot API call or CLI prompt with zero iteration. One prompt in, one HTML file out.

How Each Model Was Called

1
Claude Sonnet 4.6 & Opus 4.7 — Called via the claude CLI (claude -p) in non-interactive print mode with --dangerously-skip-permissions and --max-turns 1. This is a single-shot generation with no file access, no tool use, no iteration. Opus was selected with --model claude-opus-4-20250514. The CLI is part of a Claude Pro subscription ($20/mo).
2
GLM 5.1 — Called via the ZAI API (api.z.ai) using OpenAI-compatible chat completions endpoint. Parameters: temperature: 0.7, max_tokens: 32768, stream: true. GLM 5.1 is a reasoning model that produces thinking tokens before content. The 32K token budget was necessary because reasoning tokens consume the output budget.
3
Qwen 3.6 27B — Called via a local llama.cpp server running on a Mac Mini at 10.0.0.160:8081. Model: Qwen3.6-27B-Q4_K_M-mtp.gguf (quantized to 4-bit). Parameters: temperature: 0.7, max_tokens: 8192. 100% local inference, no data leaves the machine.
GLM Chess Note: GLM 5.1 (reasoning model) failed to generate chess after 6 attempts across multiple sessions (10+ minutes each, all reasoning tokens, zero content output). The chess output shown for GLM was generated by glm-5 (non-reasoning variant of the same family). This is noted as a legitimate finding -- reasoning models can get stuck on complex game logic tasks.

Scoring System

Each test was scored on 14 automated criteria. Scoring is based purely on code analysis -- no human judgment, no subjective quality ratings. A Python script checks each output file for specific patterns and features.

CriterionPointsDetection Method
Code Volume0-3<5KB=0, 5-15KB=1, 15-25KB=2, 25KB+=3
Dark Theme0-1Dark background colors detected in CSS
Responsive0-1@media queries present
Animations0-1transition + @keyframes/animation
SVG Usage0-1<svg> elements in output
Canvas0-1<canvas> element in output
LocalStorage0-1localStorage API used
Keyboard Support0-1keydown/keyboard event listeners
Touch Support0-1touchstart/touchmove/swipe handlers
CSS Variables0-1More than 5 custom properties (--var)
Code Organization0-1More than 10 functions or 15 arrow functions
Comments0-1More than 10 code comments

Max score per test: 14. Max total: 126 (14 × 9 tests).

Scoring bias note: This scoring system rewards feature completeness and code volume. Models that produce more code with more features score higher. It does not measure code correctness, UI polish, or user experience. A model writing 40KB of broken code could outscore one writing 10KB of perfect code. Keep this in mind when interpreting results.

Why did Claude models produce "28-byte garbage"? The claude -p CLI sometimes outputs a brief text acknowledgment instead of code when it decides to "think" about the task rather than generate. This happened 4 times with Opus and 1 time with Sonnet. All retries succeeded.

Post-Processing

The only post-processing applied to model outputs:

1
Markdown wrapper removal — If the model wrapped output in ```html ... ```, the wrappers were stripped.
2
HTML extraction — If the model included text before/after the HTML, only the content between <!DOCTYPE html> and </html> was kept.
3
Qwen chess bug fix — One line added: renderBoard() call inside initGame(). The model defined the function but forgot to call it on init.

No other edits were made to any model output. What you see is what the model generated.
