Methodology

Overview

4 AI models were given the exact same prompt for each of 9 coding tasks. Each model produced a single self-contained HTML file. No edits, no fixes, no follow-up prompts. The raw output is what you see on this site.

Claude Code CLI was used in single-shot print mode only. The claude -p command with --max-turns 1 means: prompt in, text out, no tools, no file access, no iteration, no debugging. This is functionally identical to calling the Anthropic API directly. No agentic capabilities (file editing, terminal access, multi-turn reasoning) were used. GLM and Qwen were called via direct API endpoints.

How Each Model Was Called

1
Claude Sonnet 4.6 & Opus 4.7 — Called via the Claude Code CLI in print mode: claude -p "prompt" --dangerously-skip-permissions --max-turns 1. The -p flag means non-interactive (print mode). --max-turns 1 prevents any iteration or tool use. The model receives the prompt and outputs raw text — no file access, no terminal, no agentic behavior. Opus was selected with --model claude-opus-4-20250514. Part of a Claude Pro subscription ($20/mo).
2
GLM 5.1 — Called via the ZAI API (api.z.ai) using OpenAI-compatible chat completitions endpoint. Parameters: temperature: 0.7, max_tokens: 32768, stream: true. GLM 5.1 is a reasoning model that produces thinking tokens before content. The 32K token budget was necessary because reasoning tokens consume the output budget.
3
Qwen 3.6 27B — Called via a local llama.cpp server running on a Mac Mini at 10.0.0.160:8081. Model: Qwen3.6-27B-Q4_K_M-mtp.gguf (quantized to 4-bit). Parameters: temperature: 0.7, max_tokens: 8192. 100% local inference, no data leaves the machine.
GLM Chess Note: GLM 5.1 (reasoning model) failed to generate chess after 6 attempts across multiple sessions (10+ minutes each, all reasoning tokens, zero content output). The chess output shown for GLM was generated by glm-5 (non-reasoning variant of the same family). This is noted as a legitimate finding — reasoning models can get stuck on complex game logic tasks.

Scoring System

Inspired by industry benchmarks like SWE-bench, HumanEval, BigCodeBench, and Aider Polyglot, scoring is based on functional correctness — does the code actually work? — not on code volume or feature count.

How industry benchmarks work: SWE-bench runs existing unit tests and checks % resolved. HumanEval uses pass@k — does the code pass automated test cases? BigCodeBench uses calibrated Pass@1. Aider runs 225 Exercism exercises and checks test cases pass. All of them run the code. None of them measure lines of code.

Functional Tests (0-10 per test)

Each test has 10 automated test cases — JavaScript assertions loaded and evaluated in a headless Chromium browser via Playwright. Each test case checks one specific requirement from the prompt. Pass = 1 point, Fail = 0.

Example: For the Calculator test, one assertion is document.querySelectorAll('button').length >= 10 — does the output have at least 10 buttons? Another is document.body.innerHTML.match(/[+\-*/=]/) !== null — does it have arithmetic operators?

Speed Bonus (0-2 per test)

Within each test, models are ranked by generation time. Fastest gets +2.0, second +1.5, third +1.0, slowest +0.5. Failed generations get 0.

Combined Score

Functional (0-10) + Speed Bonus (0-2) = Combined per test (0-12)

Total: 108 points maximum (12 × 9 tests)

Test Prompts

Every model received the exact same prompt for each test. The only instruction was to produce a single HTML file — no explanations, no markdown wrapping. All prompts ended with "Output ONLY raw HTML starting with <!DOCTYPE html>."

Add new tasks with title, description, priority (Low/Medium/High). Drag and drop between columns. Dark theme with CSS variables. LocalStorage persistence. Search/filter tasks. Responsive design. Task count per column. Timestamps. Delete and edit tasks. Animations on drag. Output ONLY raw HTML starting with .

Create a complete analytics dashboard as a single HTML file (inline CSS + JS, no external dependencies). Requirements: KPI cards with animated counters (Revenue $847K, Users 12.4K, Orders 3.2K, Growth +24%). Revenue chart using SVG/Canvas. Recent activity feed (10 items). Dark theme. Responsive grid layout. Auto-refresh timer (simulated data updates). Date range filter. Export button. Hover effects on cards. Output ONLY raw HTML starting with .
♟ Chess Game Game

Output ONLY raw HTML starting with .

Post-Processing

1
Markdown wrapper removal — If the model wrapped output in ```html ... ```, the wrappers were stripped.
2
HTML extraction — If the model included text before/after the HTML, only the content between <!DOCTYPE html> and </html> was kept.
3
Qwen chess bug fix — One line added: renderBoard() call inside initGame(). The model defined the function but forgot to call it on init.
